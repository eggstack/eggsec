## 1. Executive Summary

Phase 2 aims to significantly increase the value of dynamic mobile runs by adding **network visibility** and **runtime behavior validation**.

**Primary Focus Areas (P0)**:
1. **Proxy / MITM Integration** — Enable observation of real app network behavior during dynamic runs.
2. **Runtime Permission Testing** — Validate permission behavior at runtime.

**Secondary Focus Areas (P1)**:
- Static ↔ Dynamic finding correlation
- Improved evidence quality and redaction
- Report enhancements

This plan provides concrete tasks, technical approach for proxy integration (leveraging `crates/eggsec/src/proxy/`), safety considerations, and a phased delivery roadmap.

**Key Principle**: Maintain the strict standalone defense-lab model. All new capabilities remain heavily gated.

## 4. Detailed Proxy / MITM Integration Approach

### 4.1 Integration Strategy with Existing Proxy Module

The existing `proxy` module is designed for **outbound** proxied connections from Eggsec itself. For mobile dynamic testing, we need a different pattern:

**Recommended Approach (Pragmatic & Safe)**:

**Level 1 (Phase 2a – Recommended starting point)**:
- Provide high-quality **device proxy configuration** via ADB.
- Document and optionally automate setting the device's global HTTP proxy (`settings put global http_proxy <host:port>`).
- Support both:
  - External mitmproxy (user runs it separately)
  - Eggsec's own proxy infrastructure (if user has configured proxies)
- Capture traffic **summary** by either:
  - Parsing mitmproxy log output (if user points to it)
  - Or having the user provide a traffic capture file

**Level 2 (Phase 2b)**:
- Deeper integration where Eggsec can help manage or communicate with a local mitmproxy instance.
- Structured traffic findings with endpoint correlation to static analysis.

### 4.2 Proposed New Components

```rust
// In crates/eggsec/src/mobile/
pub mod traffic;           // NEW: Traffic summary types + mitmproxy log parsing

// In dynamic.rs
pub async fn configure_device_proxy(
    device: &str,
    proxy_host: &str,
    proxy_port: u16,
) -> Result<()> { ... }

pub async fn reset_device_proxy(device: &str) -> Result<()> { ... }
```

### 4.3 Traffic Summary Types (Proposed)

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TrafficSummary {
    pub total_requests: u32,
    pub cleartext_requests: u32,
    pub unique_domains: Vec<String>,
    pub suspicious_endpoints: Vec<String>, // e.g. http:// with sensitive paths
    pub findings: Vec<DynamicMobileFinding>,
}
```

### 4.4 Integration Points

- `dynamic.rs`: Add `--proxy <host:port>` and `--reset-proxy` flags. Call device configuration before/after the run.
- `adb.rs`: Add helper methods `set_global_proxy()` and `clear_global_proxy()` using `settings put/get global`.
- `runtime.rs` or new `traffic.rs`: Parser for mitmproxy `--set log_level=info` output or HAR/JSON export.
- `DynamicMobileReport`: Add optional `traffic_summary: Option<TrafficSummary>`.

## 7. Implementation Order (Recommended)

**Phase 2a (Foundation – 2–3 weeks)**
1. Device proxy configuration via ADB (`set_global_proxy`, `clear_global_proxy`)
2. Basic `--proxy` flag + documentation for mitmproxy workflow
3. High-level traffic summary type + simple parser
4. Runtime permission grant/revoke support

**Phase 2b (Correlation & Polish – 1–2 weeks)**
1. Static ↔ dynamic correlation helpers
2. Enhanced redaction and structured evidence
3. `traffic_summary` in report + bridge updates
4. Full documentation and examples (including proxy setup)

**Phase 2c (Optional Stretch)**
- Basic mitmproxy process management (start/stop via Eggsec)
- More sophisticated traffic finding generation

## 8. Safety & Policy Considerations

- All proxy and permission features must respect the existing `EnforcementContext` gate.
- Non-dry-run proxy configuration should require `--allow-dynamic-mobile`.
- Strong emphasis on:
  - User-controlled test data only
  - Redaction of sensitive traffic
  - Clear warnings about production device risks
- Lab manifest should be able to restrict allowed proxy configurations if desired (future).

---

## 9. Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Proxy configuration complexity across Android versions | Start with global `http_proxy` setting (widely supported). Document device-specific quirks. |
| Traffic redaction quality | Begin conservative. Make redaction configurable. Focus on high-signal patterns first. |
| Scope creep (wanting full mitmproxy integration too early) | Strictly limit Phase 2a to device configuration + summary. Full process management in 2b or later. |
| Dependency on external mitmproxy | Make it optional. Core value should work even if user only uses Eggsec's proxy infrastructure. |
