# Eggsec TUI Packaged Halloy Themes Plan

## Purpose

Implement packaged Halloy-style themes for `eggsec-tui` so that:

- Halloy `.toml` theme files in the repository `themes/` directory can be packaged into the binary.
- Packaged themes are installed idempotently into the user's Eggsec theme directory.
- Theme installation is best-effort and must not block or visibly slow TUI startup.
- Themes from the user theme directory are selectable in the Settings tab theme selector.
- Eggsec ships with all packaged themes as defaults.
- `cyber red` is always available as an in-binary fallback even when theme directory reads/writes fail.

This pass should build on the current TUI theme refactor. `Theme.name` is already an owned `String`, `ThemeManager.current` is private, shell/popup rendering is mostly explicit-theme, and the settings tab already has a Theme section. This plan should add bundled theme installation and selection, not restart the earlier refactors.

## Critical behavior requirements

Theme extraction must not be a hard startup dependency.

The TUI must remain usable if:

- the user config/data directory cannot be resolved,
- the themes directory cannot be created,
- packaged theme extraction fails,
- packaged theme TOML parsing fails,
- a user theme file is malformed,
- the filesystem is read-only,
- `themes/` is missing from the source checkout when packaging is not being regenerated.

In all of those cases, Eggsec should fall back to an in-binary `cyber red` theme and log a warning. It may surface a non-fatal notification after the UI is visible, but it must not panic or abort TUI startup.

Theme installation should be idempotent. On each startup, Eggsec should ensure bundled themes are installed, but it should only write missing bundled theme files by default. It must not overwrite user-edited files unless a future explicit reset/update command is added.

Startup should not appear slower to the user. If extraction or disk IO is more than trivial, run it in a non-blocking/background task after the first frame, then refresh the theme list when complete. It is acceptable for the initial settings dropdown to show only in-memory fallback themes briefly, then update after packaged/user themes are loaded.

## Current codebase notes

`crates/eggsec-tui/src/theme/palette.rs` defines `Theme` with `name: String`, `mode`, and `ThemeColors`.

`crates/eggsec-tui/src/theme/builtin.rs` currently provides in-code built-in dark/light themes.

`crates/eggsec-tui/src/theme/manager.rs` owns theme registration and current theme selection.

`crates/eggsec-tui/src/theme/legacy.rs` still contains thread-local compatibility macros. Do not expand macro usage.

`crates/eggsec-tui/src/tabs/settings/main.rs` currently has a Theme section with `dark_mode: Checkbox` and `accent_color: Selector`. The Theme section has `max_focus_index` of 1 and focus behavior for those two controls.

The desired Settings tab UI should become a theme selector based on `ThemeManager::list_themes()` or equivalent metadata. Dark-mode/accent controls can be removed, hidden, or retained as non-authoritative compatibility only, but theme selection by name should be the primary behavior.

## Dependency plan

Add `lzma-rs` to `crates/eggsec-tui/Cargo.toml` dependencies.

Use the existing `base64` workspace dependency if already available to `eggsec-tui`; if not, add it explicitly from workspace dependencies.

Do not add a C-backed LZMA dependency. The goal is decode-only pure Rust extraction.

Do not add a tar/zip dependency unless absolutely necessary. Prefer the custom minimal archive format below.

## Packaged archive format

Use a minimal custom binary archive rather than tar/zip. The Python packager and Rust decoder must agree on this format:

```text
magic: b"SLPTHEME\0"
version: u16 little-endian, currently 1
file_count: u32 little-endian
repeat file_count times:
  path_len: u16 little-endian
  path_utf8: [u8; path_len]
  content_len: u64 little-endian
  content: [u8; content_len]
```

Paths must be relative and normalized. Only package `.toml` files under the repository `themes/` directory. Path traversal must be rejected on decode:

- reject absolute paths,
- reject empty paths,
- reject `..` components,
- reject path separators that escape the theme directory,
- reject non-`.toml` packaged files.

For the first implementation, flatten all packaged theme files into the runtime `themes/` directory using their relative path from the repository `themes/` source directory. If subdirectories exist, preserve them only if the loader recursively scans `.toml`; otherwise require flat packaging and fail the script if subdirectories are present.

## Python packaging script

Create `scripts/package_themes.py`.

Responsibilities:

1. Read all `themes/*.toml` files from the repository root.
2. Sort files by relative path for deterministic output.
3. Build the custom binary archive format.
4. Compress with Python stdlib `lzma.compress(...)`.
5. Base64 encode with Python stdlib `base64.b64encode(...)`.
6. Emit `crates/eggsec-tui/src/theme/packaged.rs`.
7. Include metadata: package version, generated timestamp or deterministic package id, file count, and source file names.

Prefer deterministic output. Avoid embedding wall-clock timestamps unless useful. A deterministic package hash is better:

```python
sha256 = hashlib.sha256(archive_bytes).hexdigest()
```

Generated Rust should look roughly like:

```rust
// @generated by scripts/package_themes.py; do not edit by hand.

pub const PACKAGED_THEMES_VERSION: u16 = 1;
pub const PACKAGED_THEMES_FILE_COUNT: usize = 42;
pub const PACKAGED_THEMES_ARCHIVE_SHA256: &str = "...";
pub const PACKAGED_THEMES_LZMA_BASE64: &str = "...";

pub const PACKAGED_THEME_FILES: &[&str] = &[
    "cyber-red.toml",
    ...
];
```

The script should fail with a clear error if:

- `themes/` is missing,
- no `.toml` files are found,
- duplicate relative paths are found,
- `cyber red` cannot be identified,
- a packaged file path is unsafe.

The script should support at least:

```bash
python3 scripts/package_themes.py
```

Optionally support:

```bash
python3 scripts/package_themes.py --themes-dir themes --out crates/eggsec-tui/src/theme/packaged.rs
```

## Cyber Red fallback requirement

`cyber red` must be available even if the themes directory cannot be read or written.

Implement this by embedding a native in-code `cyber_red_theme()` in `theme/builtin.rs`, not solely by relying on packaged TOML extraction. This is important because packaged extraction still depends on base64 decode, LZMA decode, and possibly theme TOML parsing.

`ThemeManager::new()` or the new default constructor should register at least:

- `cyber red` / canonical id `cyber-red`,
- existing `dark`,
- existing `light` if desired.

Set `cyber-red` as the default current theme.

If a packaged/user theme also has id/name `cyber-red`, the manager should either:

- keep the native fallback as authoritative and skip duplicate registration, or
- allow file-loaded theme to replace it only after successful parsing.

For safety and predictability, prefer keeping native `cyber-red` authoritative unless there is a strong reason to override it.

Acceptance criteria:

- `ThemeManager::new().current().name` is `cyber-red` or a clearly canonical equivalent.
- The TUI can launch with only in-binary themes.
- Theme directory failures do not prevent the Cyber Red fallback.

## Theme directory resolution

Use the existing `directories` dependency. Add a helper in `theme/install.rs` or `theme/loader.rs`:

```rust
pub fn user_theme_dir() -> Option<PathBuf>
```

Use a platform-appropriate config/data path, e.g. via `ProjectDirs`:

```text
Linux:   $XDG_CONFIG_HOME/eggsec/themes or ~/.config/eggsec/themes
macOS:   ~/Library/Application Support/eggsec/themes
Windows: %APPDATA%/eggsec/themes
```

Keep repository `themes/` as source input for the packaging script only. Runtime should not assume the repository root exists.

If the directory cannot be resolved, return a non-fatal error and continue with in-memory themes.

## Non-blocking theme installation/loading

Startup behavior should be staged:

1. Construct `App` with `ThemeManager::new()` using in-binary themes, defaulting to Cyber Red.
2. Start/render TUI normally.
3. Kick off best-effort packaged theme installation and user theme directory loading in a background Tokio task, or do it before first frame only if proven trivial.
4. When loading completes, merge discovered themes into `ThemeManager`, update Settings tab selector items, and preserve the current theme unless a restored session/config theme becomes available.
5. If loading fails, keep Cyber Red and log a warning.

A simpler first implementation may run ensure/load synchronously if the code is very fast, but do not allow it to block on slow filesystems. Prefer the async/background design if the TUI runner already has task/event plumbing.

Suggested result type:

```rust
pub struct ThemeInstallReport {
    pub theme_dir: Option<PathBuf>,
    pub installed: usize,
    pub skipped_existing: usize,
    pub loaded: usize,
    pub errors: Vec<String>,
}
```

The report can be logged and optionally surfaced as a low-severity notification after the UI is visible.

## Theme loader design

Add `theme/loader.rs` to parse Halloy `.toml` into Eggsec `Theme`.

Do not deserialize directly into `ThemeColors` unless Halloy schema exactly matches Eggsec. Use an intermediate struct with permissive optional fields.

The loader should:

- parse TOML using the existing `toml` dependency,
- identify a canonical theme id from file stem or a name field,
- parse colors from strings such as `#RRGGBB`, `#RGB` if present, and basic named colors if needed,
- map Halloy palette fields into Eggsec semantic colors,
- fill missing Eggsec-specific colors via fallback defaults.

Suggested fallback strategy:

- Start from `cyber_red_theme()` or `dark_theme()` depending on the detected background/mode.
- Override semantic colors where Halloy values exist.
- Preserve status colors using sensible fallback values if Halloy does not define success/warning/error.

The loader must never panic on a bad theme. Return a structured error for that file and continue loading other files.

## Theme manager extensions

Extend `ThemeManager` with registration and directory loading methods. Suggested API:

```rust
impl ThemeManager {
    pub fn register_theme(&mut self, theme: Theme) -> Result<(), ThemeError>;
    pub fn register_theme_if_absent(&mut self, theme: Theme) -> bool;
    pub fn load_from_dir_best_effort(&mut self, dir: &Path) -> ThemeLoadReport;
    pub fn list_theme_items(&self) -> Vec<ThemeListItem>;
}

pub struct ThemeListItem {
    pub id: String,
    pub label: String,
    pub source: ThemeSource,
}

pub enum ThemeSource {
    Builtin,
    Packaged,
    UserFile(PathBuf),
}
```

If a smaller implementation is needed, keep `list_themes()` returning `Vec<&str>` and add only `register_theme`. However, settings UI will be cleaner with display labels and stable ids.

Duplicate names should be handled deterministically. Prefer this precedence:

1. Built-in Cyber Red fallback,
2. user theme files,
3. packaged installed theme files,
4. other built-ins.

But do not let malformed user themes remove a working built-in.

## Settings tab integration

Replace the current Theme section controls with a theme selector.

Current controls:

- `dark_mode: Checkbox`,
- `accent_color: Selector`.

Desired control:

- `theme_selector: Selector` with items populated from `ThemeManager`.

Implementation guidance:

1. Add `theme_selector: Selector` to `SettingsTab`.
2. Add a method:

```rust
pub fn set_available_themes(&mut self, themes: &[ThemeListItem], current_theme: &str)
```

or, if using names only:

```rust
pub fn set_available_themes<I>(&mut self, themes: I, current_theme: &str)

```

3. Update `sync_with_theme(&Theme)` so it selects the current theme by name/id rather than inferring dark mode/accent color.
4. Update `max_focus_index` for `SettingsSection::Theme` to match the new number of controls, likely `0` if only one selector exists.
5. Update focus management to keep the theme selector open when focused.
6. Update rendering in `tabs/settings/render.rs` to show the selector and a short help line such as:

```text
Theme files are loaded from <theme-dir>. Bundled themes are installed automatically when possible.
```

7. Update key handling so selecting a theme applies it to `app.theme_manager`, syncs legacy thread-local theme state, syncs the settings tab, and marks `needs_redraw = true`.

If config/session persistence already stores `theme_name`, reuse that path. If not, store selected theme in the TUI session state first; do not invent a broad config migration unless needed.

## App startup integration

In `App::new_inner`, initialize with in-memory `ThemeManager::new()` first. Cyber Red should be current immediately.

Then schedule or perform best-effort theme install/load:

- synchronous only if trivial and bounded,
- otherwise background task with report/event back to App.

If current session restore contains a theme name:

1. Try to apply it against currently registered built-ins.
2. After packaged/user themes load, try again if it was not initially available.
3. If still unavailable, keep Cyber Red and log/notify non-fatally.

Add an `Option<String>` field if needed to remember a deferred restored theme name until loading completes.

Update settings selector after themes load.

## Archive decode/install implementation details

Add `theme/archive.rs` decode helpers:

```rust
pub struct PackagedThemeFile {
    pub relative_path: PathBuf,
    pub content: Vec<u8>,
}

pub fn decode_packaged_archive(bytes: &[u8]) -> Result<Vec<PackagedThemeFile>, ThemeArchiveError>
```

Add `theme/install.rs`:

```rust
pub fn decode_packaged_themes() -> Result<Vec<PackagedThemeFile>, ThemeInstallError>
pub fn ensure_packaged_themes_installed(dir: &Path) -> ThemeInstallReport
```

Use:

- `base64` decode,
- `lzma_rs::lzma_decompress` or the matching format used by Python stdlib `lzma.compress`.

Important: Python `lzma.compress(...)` defaults to XZ container format. In Rust, use the corresponding `lzma_rs::xz_decompress(...)`, not raw `lzma_decompress`, unless the Python script explicitly emits raw LZMA. Prefer XZ container for simplicity and metadata integrity.

Writing files:

- Create the themes directory if missing.
- For each packaged file, if destination exists, skip.
- If destination is missing, write atomically where practical: write `filename.tmp`, flush, then rename.
- Reject unsafe paths even if generated by our script.
- Collect errors but do not fail the entire TUI.

## Acceptance criteria

This pass is complete when:

- `scripts/package_themes.py` generates `theme/packaged.rs` from repository `themes/*.toml`.
- Packaged themes decode with `lzma-rs` using pure Rust decode.
- Bundled themes are installed idempotently into the user themes directory when possible.
- Failure to install or load themes does not block TUI startup.
- Cyber Red is always available in binary and is the default fallback.
- The Settings tab has a theme selector populated from available registered themes.
- Selecting a theme from Settings applies immediately and persists through the existing session/config path if present.
- Existing dark/light themes either remain as built-ins or are intentionally superseded by packaged themes.
- The code logs non-fatal theme install/load errors clearly.
- Tests cover archive safety, installer no-overwrite behavior, fallback behavior, and settings selector behavior.

## Implementation notes for smaller models

Keep the implementation staged and compile after each phase.

Do not make theme install/load part of a hard `App::new()` success path.

Do not panic on malformed user theme files.

Do not overwrite existing theme files.

Do not assume the repository root exists at runtime.

Do not remove Cyber Red from built-ins even if a packaged Cyber Red TOML exists.

Prefer simple explicit code over a generic plugin-style theme system. The goal is a robust bundled theme experience, not an extensibility framework.
