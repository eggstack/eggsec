# C2 (Command & Control) Module

## Overview

The C2 module provides a lightweight Rust-native Command & Control framework for defense-lab purple teaming and red team simulation. It enables realistic C2 operations including beaconing, tasking, campaign orchestration, and OPSEC assessment.

**Feature gate**: `c2` (depends on `postex` and `evasion`)
**Mode**: Defense Lab only
**Safety**: Dry-run default; real execution requires `--allow-c2` policy flag

## Architecture

### Module Structure

```
c2/
├── mod.rs          # Core types, C2Scanner, to_scan_report_data, run_cli
├── beacon.rs       # Beacon protocol simulation (HTTP/S, DNS, TCP)
├── tasking.rs      # Task queue and execution simulation
├── campaign.rs     # Campaign profile catalog
└── opsec.rs        # OPSEC scoring and anti-forensics assessment
```

### Key Types

| Type | Description |
|------|-------------|
| `C2Report` | Top-level report containing campaign, beacons, tasks, and OPSEC assessment |
| `C2Campaign` | Campaign definition with MITRE ATT&CK profile and phases |
| `CampaignPhase` | Individual phase within a campaign with MITRE technique mappings |
| `BeaconResult` | Result of a beacon attempt (protocol, timing, success) |
| `BeaconProtocol` | Enum of supported beacon protocols (Http, Https, Dns, Tcp, Custom) |
| `TaskResult` | Result of a C2 task (type, status, output) |
| `TaskType` | Enum of task types (Recon, Execute, Exfil, Persist, Lateral, Evade, SelfDestruct) |
| `TaskStatus` | Enum of task statuses (Completed, Failed, Simulated, Denied) |
| `OpsecAssessment` | OPSEC scoring with findings and recommendations |
| `OpsecFinding` | Individual OPSEC finding with category and severity |
| `OpsecCategory` | Enum of OPSEC categories (ParentSpoofing, Timestomping, etc.) |
| `OpsecSeverity` | Severity level for OPSEC findings (Info, Low, Medium, High) |
| `C2Scanner` | Main scanner struct with dry-run support |

### Policy Integration

- **OperationRisk::C2Operation** - Risk tier for real C2 operations
- **ExecutionPolicy::allow_c2_operations** - Policy flag controlling C2 access
- **Dry-run**: Risk reduced to `SafeActive` (always allowed)
- **Real**: Risk elevated to `C2Operation` (requires policy flag)

### Campaign Profiles

| Profile | MITRE | Description |
|---------|-------|-------------|
| `apt29` | APT29 (Cozy Bear) | HTTP/S beacons, LOTL techniques, 4-phase campaign |
| `carbanak` | Carbanak/FIN7 | DNS beacons, financial targeting, 3-phase campaign |
| `default` | Generic APT | Mixed protocols, 2-phase campaign |

### Reporting Bridge

The C2 module provides a `to_scan_report_data()` bridge function that converts `C2Report` to the unified `ScanReportData` format. This bridge is wired into the `report convert` handler for auto-detection.

Categories: `c2-beacon`, `c2-task-*`, `c2-parent-spoofing`, `c2-timestomping`, `c2-log-tampering`, `c2-process-masquerading`, `c2-burn-mechanism`, `c2-decoy-activity`, `c2-summary`

## Usage

### CLI

```bash
# Dry-run with default campaign
eggsec c2 --target 10.0.0.1 --dry-run --json

# Dry-run with APT29 profile
eggsec c2 --target 10.0.0.1 --campaign apt29 --dry-run

# Output to file
eggsec c2 --target 10.0.0.1 --campaign carbanak --dry-run -o c2-report.json

# Convert to SARIF
eggsec report convert c2-report.json -f sarif
```

### Policy Configuration

```toml
[execution_policy]
allow_c2_operations = true  # Required for non-dry-run C2 operations
```

## Safety

- Dry-run mode produces complete reports with synthetic data (no real C2 operations)
- Real execution requires explicit `--allow-c2` policy flag
- All operations are scoped to authorized lab environments
- OPSEC assessment provides defensive recommendations
- Standalone defense-lab surface (no MCP/agent/TUI/pipeline integration)
