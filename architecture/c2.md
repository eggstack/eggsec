# C2 (Command & Control) Module

## Overview

The C2 module provides a lightweight Rust-native Command & Control framework for defense-lab purple teaming and red team simulation. It enables realistic C2 operations including beaconing, tasking, campaign orchestration, and OPSEC assessment.

**Feature gate**: `c2` (depends on `postex` and `evasion`)
**Mode**: Defense Lab only
**Safety**: Dry-run default; real execution requires `--allow-c2` policy flag

## Architecture

### Module Structure

```
c2/
├── mod.rs          # Core types, C2Scanner, to_scan_report_data, run_cli
├── agent.rs        # Agent lifecycle (register, check-in, self-destruct, state machine)
├── beacon.rs       # Beacon protocol simulation (HTTP/S, DNS, TCP)
├── tasking.rs      # Task queue with postex integration (LOTL, lateral, credential, persistence)
├── campaign.rs     # Campaign profiles, attack graph generation, timeline
└── opsec.rs        # OPSEC scoring and anti-forensics assessment
```

### Key Types

| Type | Description |
|------|-------------|
| `C2Report` | Top-level report containing campaign, beacons, tasks, OPSEC, attack graph, and timeline |
| `C2Campaign` | Campaign definition with MITRE ATT&CK profile and phases |
| `CampaignPhase` | Individual phase within a campaign with MITRE technique mappings |
| `BeaconResult` | Result of a beacon attempt (protocol, timing, success) |
| `BeaconProtocol` | Enum of supported beacon protocols (Http, Https, Dns, Tcp, Custom) |
| `TaskResult` | Result of a C2 task (type, status, output) |
| `TaskType` | Enum of task types (Recon, Execute, Exfil, Persist, Lateral, Evade, SelfDestruct) |
| `TaskStatus` | Enum of task statuses (Completed, Failed, Simulated, Denied) |
| `AgentConfig` | Agent configuration (beacon interval, jitter, max check-ins, self-destruct) |
| `AgentInstance` | Agent runtime state (state machine, check-in count, task counters) |
| `AgentState` | Agent lifecycle state (Registered, Active, Idle, Compromised, SelfDestructed, Terminated) |
| `AgentLifecycleResult` | Full agent lifecycle simulation result with events and tasks |
| `AttackGraph` | Campaign dependency graph with nodes and critical path |
| `AttackGraphNode` | Single technique in the graph with phase and dependency info |
| `CampaignTimeline` | Sequential event timeline with phase progression |
| `TimelineEntry` | Single event in the timeline with timestamp and dependencies |
| `OpsecAssessment` | OPSEC scoring with findings and recommendations |
| `OpsecFinding` | Individual OPSEC finding with category and severity |
| `OpsecCategory` | Enum of OPSEC categories (ParentSpoofing, Timestomping, etc.) |
| `OpsecSeverity` | Severity level for OPSEC findings (Info, Low, Medium, High) |
| `C2Scanner` | Main scanner struct with dry-run support |

### Policy Integration

- **OperationRisk::C2Operation** - Risk tier for real C2 operations
- **ExecutionPolicy::allow_cateral** - Policy flag controlling C2 access
- **Dry-run**: Risk reduced to `SafeActive` (always allowed)
- **Real**: Risk elevated to `C2Operation` (requires policy flag + `--allow-c2`)
- **MCP**: `c2-mcp` marker feature exposes C2 tool to MCP server (always forces dry-run)
- **TUI**: Always runs dry-run (no dry-run toggle in TUI task builder)

### Dry-Run vs Real Mode

The C2 module follows the same pattern as db-pentest and wireless active:

| Entry Point | Dry-Run Behavior | Real Behavior | Gating |
|-------------|-----------------|---------------|--------|
| CLI (`eggsec c2 --dry-run`) | Synthetic beacons, tasks, OPSEC | Real HTTP/S beacons, TCP port scans, HTTP task delivery | `--allow-c2` + `OperationRisk::C2Operation` |
| TUI (`Tab::C2`) | Synthetic (always) | N/A (TUI always dry-run) | N/A |
| MCP (`c2` tool) | Synthetic (always) | N/A (MCP always dry-run) | `c2-mcp` feature |

**Dry-run path** (`dry_run_simulation`):
- `beacon::simulate_beacons()` — pure synthetic results, no I/O
- `tasking::simulate_tasks()` — pure synthetic results, no I/O
- `opsec::simulate_opsec_assessment()` — always synthetic (auditing is data-only)

**Real path** (`real_simulation`):
- `beacon::simulate_beacons()` — real HTTP/S requests via `reqwest` (10s timeout), DNS resolution, TCP connect
- `tasking::simulate_tasks()` — TCP port scan (80, 443, 8443, 8080, 4443), HTTP POST task delivery, 1KB exfil payload, decoy traffic
- `opsec::simulate_opsec_assessment()` — always synthetic (same as dry-run)

**Safety**: All real network I/O has timeouts (10s beacons, 5s tasks, 3s port scans). Connection failures produce `success: false` / `TaskStatus::Failed` with error evidence. No panics. No actual attack primitives executed.

### Campaign Profiles

| Profile | MITRE | Description |
|---------|-------|-------------|
| `apt29` | APT29 (Cozy Bear) | HTTP/S beacons, LOTL techniques, 4-phase campaign |
| `carbanak` | Carbanak/FIN7 | DNS beacons, financial targeting, 3-phase campaign |
| `default` | Generic APT | Mixed protocols, 2-phase campaign |

### Reporting Bridge

The C2 module provides a `to_scan_report_data()` bridge function that converts `C2Report` to the unified `ScanReportData` format. This bridge is wired into the `report convert` handler for auto-detection.

Categories: `c2-beacon`, `c2-task-*`, `c2-parent-spoofing`, `c2-timestomping`, `c2-log-tampering`, `c2-process-masquerading`, `c2-burn-mechanism`, `c2-decoy-activity`, `c2-summary`

### MCP Tool Exposure

C2 MCP tool is gated behind the `c2-mcp` marker feature (requires `c2`):

```toml
[dependencies]
eggsec = { features = ["c2-mcp"] }
```

The MCP tool (`c2`) is registered in `create_default_registry()` and exposed to:
- **OpsAgent**: visible (uses `ToolSelector::All`)
- **CodingAgent**: hidden (explicit allowlist does not include c2)

Tool parameters:
- `target` (optional, default: "localhost") - Target for simulation
- `campaign` (optional, default: "default") - Campaign profile (apt29, carbanak, default)

**Safety**: Always forces `dry_run: true` regardless of user input. The MCP pre-dispatch enforcement gate validates scope and policy before the tool is reached.

## Usage

### CLI

```bash
# Dry-run with default campaign
eggsec c2 --target 10.0.0.1 --dry-run --json

# Dry-run with APT29 profile
eggsec c2 --target 10.0.0.1 --campaign apt29 --dry-run

# Real C2 simulation (requires --allow-c2)
eggsec c2 --target 10.0.0.1 --allow-c2 --json

# Real C2 with APT29 profile
eggsec c2 --target 10.0.0.1 --campaign apt29 --allow-c2

# Output to file
eggsec c2 --target 10.0.0.1 --campaign carbanak --dry-run -o c2-report.json

# Convert to SARIF
eggsec report convert c2-report.json -f sarif
```

### Policy Configuration

```toml
[execution_policy]
allow_c2_operations = true  # Required for non-dry-run C2 operations
```

## Safety

- Dry-run mode produces complete reports with synthetic data (no real C2 operations)
- Real execution requires explicit `--allow-c2` policy flag
- All operations are scoped to authorized lab environments
- OPSEC assessment provides defensive recommendations
- Standalone defense-lab surface with TUI tab `Tab::C2` (always runs dry-run); MCP exposure via `c2-mcp` marker feature

## Purple-Team Usage Examples

### Example 1: Dry-Run APT29 Campaign (Planning Phase)

```bash
# Plan an APT29 campaign against lab environment
eggsec c2 --target 10.0.0.0/24 --campaign apt29 --dry-run --json -o apt29-plan.json

# Review attack graph and timeline
cat apt29-plan.json | jq '.attack_graph.critical_path'
cat apt29-plan.json | jq '.timeline.entries[] | "\(.phase_name): \(.technique_id)"'

# Convert to SARIF for CI integration
eggsec report convert apt29-plan.json -f sarif -o apt29-plan.sarif
```

### Example 2: Postex Integration

The C2 tasking module automatically maps postex techniques to C2 tasks:
- **LOTL techniques** (T1218.011, T1105, etc.) → `TaskType::Execute` with postex enrichment
- **Lateral movement** (T1021.002, T1021.001, etc.) → `TaskType::Lateral`
- **Credential access** (T1003, T1555) → `TaskType::Recon` (completed status)
- **Persistence** (T1547.001, T1547.002) → `TaskType::Persist`

### Example 3: Agent Lifecycle Simulation

```bash
# Simulate an agent with auto-self-destruct after 50 check-ins
eggsec c2 --target 10.0.0.1 --campaign default --dry-run --json | jq '.attack_graph'
```

The agent lifecycle produces a timeline of events:
1. Registered → check-ins across campaign phases → SelfDestructed or Terminated
2. Each phase generates tasks mapped to MITRE techniques
3. OPSEC scoring tracks evasion quality across the campaign

### Example 4: Attack Graph Analysis

The attack graph connects techniques across phases:
- Phase 1 techniques have no dependencies
- Phase 2+ techniques depend on the previous phase's techniques
- Critical path highlights the primary attack chain

```json

  "critical_path": ["T1071.001", "T1547.001", "T1021.002", "T1041"],

    { "technique_id": "T1021.002", "phase_name": "Lateral Movement", "depends_on": ["T1547.001", "T1070.006"] }
  ]
}
```

### Example 5: OPSEC Assessment

The OPSEC scoring evaluates six categories:
- **ParentSpoofing**: Process hierarchy masking
- **Timestomping**: File timestamp manipulation
- **LogTampering**: Log entry removal/modification
- **ProcessMasquerading**: Legitimate process impersonation
- **BurnMechanism**: Evidence destruction capability
- **DecoyActivity**: Misdirection and noise generation

Each finding includes a severity and defensive recommendation.
